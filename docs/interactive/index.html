The Platform

Aegis — Access Control for Multi-Tenant Microservices

Aegis is an enterprise-grade access-control system for a multi-tenant, microservices-based SaaS platform: centralized authorization (RBAC + ABAC via an explicit PDP / PEP split), database-enforced tenant isolation with PostgreSQL Row-Level Security, secure service-to-service communication, dynamic role/permission management, and tamper-evident auditing — applied uniformly across seven business services.

Authorization

One shared @aegis/access-control library. Every route is authenticate → authorize(permission) → handler. Decisions are pure, cacheable, and fail-closed.

Isolation

Shared schema + tenant_id on every row + Postgres RLS (FORCE ROW LEVEL SECURITY, restrictive policy, app runs as a non-owner). Belt-and-suspenders.

Trust

Edge JWT validation at the gateway, re-validated per service via JWKS; signed internal JWTs + context propagation; hash-chained audit on every write.

Shared substrate. All eight apps sit on the same libraries — @aegis/service-core (request context, logger, HTTP client, middleware), @aegis/access-control (the PDP + PEP guards), @aegis/connectors (pluggable ERP framework), @aegis/db (Sequelize + RLS helpers), @aegis/events (the bus), plus @aegis/shared-enums / shared-types / shared-constants. A new service inherits the whole security posture for free.

System Topology

Architecture

A single edge gateway routes to seven business services. Each service is the same shape (Express + InversifyJS), and each leans on the shared libraries below it. The data plane is PostgreSQL with RLS, Redis for cache/queues, and an abstracted event bus.

PDP / PEP · RBAC + ABAC + Tenant Isolation

Access Control

A PEP (the authorize() guard inside a service) loads the resource and calls the PDP in @aegis/access-control. The PDP runs a deterministic, fail-closed pipeline: tenant isolation → RBAC permission check → ABAC conditions → obligations. Try it.

PDP.decide(principal, action, resource, context) → { allow, reason, obligations }

Permissions × roles matrix

RBAC core: which roles carry which dotted domain.action permissions. ABAC conditions (ownership, amount thresholds, segregation of duties) then narrow a granted permission at decision time.

Fail-closed everywhere. If the principal's tenant does not match the resource's tenant, the PDP denies before RBAC even runs — and the DB would reject the row anyway under RLS. Missing attributes, an unreachable PDP, or an expired decision-cache entry all resolve to DENY.

Row-Level Security

Multi-Tenancy

Aegis runs the pool model: one shared schema, tenant_id on every row, and Postgres RLS as the backstop. The app connects as a non-owner role without BYPASSRLS; each transaction begins with SET LOCAL app.current_tenant. An application bug cannot leak across tenants because the database itself filters.

The policy (once per table)

-- app role is NOT the table owner and has no BYPASSRLS
ALTER TABLE expense_reports ENABLE ROW LEVEL SECURITY;
ALTER TABLE expense_reports FORCE  ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON expense_reports
  AS RESTRICTIVE
  USING (tenant_id = current_setting('app.current_tenant')::uuid)
  WITH CHECK (tenant_id = current_setting('app.current_tenant')::uuid);
-- per request, inside the transaction (rolls back at COMMIT)
SET LOCAL app.current_tenant = '<tenant from verified JWT>';
SELECT * FROM expense_reports;  -- silently scoped
SET LOCAL, never SET. Under transaction-mode pooling, a bare SET persists on the pooled connection and the next request inherits the previous tenant. SET LOCAL (or set_config(...,true)) is mandatory.

See it filter

The same physical table holds rows for two tenants. Switch the session tenant — the rows that SET LOCAL would hide are removed from the result set, exactly as RLS does in the DB.

SET LOCAL app.current_tenant =
tenant
id
title
amount

Belt-and-suspenders. The JWT carries tenant_id, the PDP rejects cross-tenant resources, the repository compiles a tenant_id = predicate, and RLS enforces it in the engine. Four independent layers; any one failing does not leak data. Payroll is the first candidate for a hard-isolation silo tier (schema- or DB-per-tenant) for regulated customers.

Pluggable Integration

ERP Connectors

@aegis/connectors is a pluggable framework: a common connector interface, an adapter/strategy per ERP, and a registry. Adding a new ERP means writing one adapter — nothing else changes. Aegis ships mock connectors that emulate the handshake / push / status behavior without calling real systems.

Pick a registered connector

The interface every adapter implements

export interface ErpConnector {
  readonly key: ConnectorKey;            // LEDGER_ONE | FINOVO | ACCT_BRIDGE
  authenticate(cfg: ConnectorConfig): Promise<ConnectorSession>;
  pushTransaction(
    txn: OutboundTransaction,
    opts: { idempotencyKey: string }       // dedupe at the boundary
  ): Promise<PushResult>;
  fetchStatus(externalRef: string): Promise<SyncStatus>;
}
Same security path as any internal call. Connector calls flow through the service-to-service auth + request-context propagation + secret-proxy patterns. Every push carries an idempotency key, so a retry (or a duplicate push) is recognized and returns the original result instead of double-posting. Outbound auth is per-connector, carried via the connector's configured scheme — there is no shared tracking header.

Build Once, Promote Forward

Deployment

One multi-stage image builds the whole monorepo. A single entrypoint switches its role by PROCESS_TYPE. Images are immutable and tagged by Git SHA — promotion across environments is a re-tag, never a rebuild, so what you tested is byte-for-byte what ships.

One image, three roles — PROCESS_TYPE

PROCESS_TYPE=api
Boots the app named by SERVICE_NAME (e.g. expense): Express + Inversify, readiness-gated.
PROCESS_TYPE=worker
Consumes the event bus / BullMQ queues — report exports, scheduled workflow actions, async notifications.
PROCESS_TYPE=migration
Runs Umzug migrations as a one-shot task using the same image, then exits.
# scripts/start.sh — the entrypoint switch
case "$PROCESS_TYPE" in
  api)       exec node dist/apps/"$SERVICE_NAME"/main.js ;;
  worker)    exec node dist/apps/worker/main.js ;;
  migration) exec node dist/apps/cli/main.js migrate ;;
esac

SHA-image promotion: dev → staging → prod

CI builds and pushes aegis:<git-sha> once. Promotion re-tags the same digest — no rebuild, no drift.

Runtime secrets, never baked. The image carries no secrets. Each environment injects them from a parameter store keyed by env prefix (/aegis/<env>/...). CI is DRY (YAML anchors) and gated by nx affected; /health?details=true probes DB + cache + bus and readiness gates traffic.

One-command local run

A fresh machine, no manual env setup — Postgres (pre-seeded with the RLS non-owner app role), Redis, and every service on one Docker network with committed dummy .env values:

# brings the whole platform up, wired end-to-end
./scripts/dev-up.sh        # or the VS Code build task — Cmd+Shift+B

Per-service apps/<svc>/.env files are committed with internally-consistent dummy values so docker compose up works out of the box. Real environments override via env / param store.
